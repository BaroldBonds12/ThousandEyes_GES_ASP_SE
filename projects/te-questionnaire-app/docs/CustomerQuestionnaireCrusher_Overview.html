✨ Key Features
📂
Multi-Format File Parsing
Accepts Excel workbooks (RFP matrices), CSV files, Word documents, PDFs, and plain text. Intelligently extracts questions from cells ending in "?", dedicated "Question" columns, "Q:" prefixed lines, and structured RFP requirement rows.
.xlsx.docx .pdf.csv .txt
🌐
Live ThousandEyes Docs Fetching
For every question, the app searches docs.thousandeyes.com via DuckDuckGo (site: operator) and the MkDocs search index JSON. Up to 5 real content pages are fetched and stripped to main body text — no search stubs, no nav links — then injected as context for the LLM.
Real pages only DuckDuckGo MkDocs index
🤖
Local LLM Inference (Ollama)
Answers are generated entirely on-device using Ollama. Supports any model available via ollama pull — llama3.2, phi3:mini, llama3.1:8b, mistral, and qwen2.5:7b are recommended and pre-listed. The model selector auto-populates with whatever is installed.
No API keys Air-gapped llama3.2

Three-Tier Answer Priority
Every question passes through three resolution layers: User Corrections (permanent, fuzzy-matched at ≥80% similarity) → LLM Cache (7-day TTL, near-instant) → Fresh Inference (live doc fetch + Ollama). This minimises redundant AI calls while keeping answers up to date.

Data Flow — File Processing Sequence
What happens step-by-step from file selection to saved output
Step 1
File Selected & Parsed
file_parser.py reads the input file and extracts raw text/rows. question_extractor.py applies heuristics to identify questions, RFP requirements, sections, and categories. Each question gets a deterministic SHA-based cache key.
Step 2
Feedback & Cache Resolution
FeedbackStore is checked first — fuzzy-matches each question against stored user corrections (≥80% ratio). Then AnswerCache (SQLite) is checked for non-expired entries. Items resolved at this step skip doc fetching and LLM inference entirely.
Step 3
URL Enrichment (Cached Items)
For cached/feedback items missing a real source URL, a parallel fast lookup runs via get_source_url() — DuckDuckGo search returns the first real docs.thousandeyes.com content page URL (no full page fetch). The URL is written back to the SQLite cache for future runs.
Step 4
Parallel Doc Fetching
Remaining questions trigger a ThreadPoolExecutor (4 workers) that calls TESearcher.search_and_fetch() concurrently. Each call: DuckDuckGo → MkDocs index JSON → HTML scrape fallback. Results filtered by _is_content_url() — only real doc pages accepted. Up to 5 pages × 5,500 chars each.
Step 5
Sequential LLM Inference
Each remaining question is answered by LLMEngine via Ollama's HTTP API (localhost:11434). Context = human-reviewed Q&A pairs from qa_reference.py + fetched doc page content. The first [Source: URL] marker is extracted as the source URL. Answers and URLs are persisted to cache.
Step 6
Spreadsheet Review
Results populate a 3-column editable spreadsheet (Question | Answer | Source URL). Rows render incrementally (1 per 35ms tick) to keep the UI responsive. Every cell is editable inline — changes are committed to FeedbackStore on Save for future reuse.
Step 7
Output Generation
Clicking "Save Updated File" triggers file_writer.py: (a) a branded Word document with cover page + 4-column Q&A table, (b) a companion Excel file (Q|A|URL), and (c) an update to the persistent QA reference at ~/.te_qa_cache/qa_reference.xlsx — seeding better answers for the next run.

🔒 Security & Privacy
Zero-Exfiltration Architecture

Customer questionnaire content, question text, and generated answers never leave the local machine. The only outbound network traffic is: (a) read-only HTTP GET requests to docs.thousandeyes.com to fetch documentation, and (b) DuckDuckGo search queries (question text, no customer identifiers). All AI inference runs locally via Ollama. The app is suitable for environments with strict data residency requirements.

1
No API Keys Required
Ollama, DuckDuckGo search, and docs.thousandeyes.com scraping require no authentication credentials. Nothing to rotate, audit, or accidentally leak.
2
Content URL Allow-listing
_is_content_url() strictly allows only docs.thousandeyes.com content paths. No redirects to external sites, no fetching of arbitrary URLs, no SSRF risk.
3
Local-Only Storage
All persistent state (SQLite cache, feedback JSON, QA reference Excel) lives in ~/.te_qa_cache/. No cloud sync, no remote logging, no telemetry.
4
Polite Web Crawling
300ms inter-request delay and a realistic browser User-Agent string ensure docs.thousandeyes.com is not overloaded. In-memory page cache avoids re-fetching the same URL twice per session.
🚀 Deployment & Quick Start
1
Install Ollama
Download from ollama.com/download and run ollama pull llama3.2 (recommended) or any other supported model.
2
Launch the App
Unzip CustomerQuestionnaireCrusher_v2.zip and open TE Questionnaire Automator.app from dist/v2/. On first launch, right-click → Open to bypass Gatekeeper.
3
Select a Questionnaire File
Click the file drop zone or drag-and-drop an Excel RFP matrix, Word questionnaire, PDF, CSV, or plain text file.
4
Click ▶ Process File
Vizzy will guide you through the run. Cached questions resolve in <1s. Fresh questions take ~20-25s each (doc fetch + LLM). ETA is shown for long runs.
5
Review, Edit, and Save
Edit any answer or source URL directly in the spreadsheet. Click 💾 Save Updated File — a Word document and Excel companion are generated automatically. Corrections are stored for future runs.
6
Clear Cache When Needed
Use 🗑 Clear LLM Cache in the footer to force fresh AI answers (e.g. after a major ThousandEyes docs update). The QA reference is preserved.
